## 資料庫設計原則

### 基本規範

- **拒絕將密碼明文存入 users 表**（會被扣分）
- **Data Modeling 是核心工作**，不能遺漏或考慮不周
- 資料**不要重複儲存**（正規化）
- 將資訊拆成適當欄位，**但不要過度拆細**

### 資料型別選擇

| 資料 | 建議型別 | 原因 |
|------|----------|------|
| 電話號碼 | `TEXT` / `VARCHAR` | 可能有前導零、國碼符號 |
| 有加減乘除意義的數值 | `NUMERIC` / `INT` | 需要計算 |
| 無計算意義的代碼 | `TEXT` / `VARCHAR` | 語意是文字 |
| 時間 | 依各資料庫邏輯設計 | 存多細是設計決策 |

### 時間設計

- 各資料庫存時間的邏輯不同，需花心思設計「存多細」的問題

## 外鍵與刪除行為

### 刪除的連帶問題

設定外鍵後，刪除主表資料時需考慮：

- 若商品（或車次等）被刪除，關聯訂單怎麼辦？
- 若使用者刪除帳號，歷史訂單、消費紀錄如何處理？

### 軟刪除 vs 硬刪除

| | 軟刪除（Soft Delete） | 硬刪除（Hard Delete） |
|---|---|---|
| 做法 | 加 `is_active = FALSE` 欄位 | 直接 `DELETE` 資料列 |
| 優點 | 保留歷史資料 | 資料庫乾淨 |
| 缺點 | 查詢需過濾 | 資料無法復原 |

**建議**：對有歷史紀錄需求的資料（使用者、訂單）使用**軟刪除**，查詢時加 `WHERE is_active = TRUE`。

## 期末專案規格

### 資料庫要求

- **至少三種資料庫**
  - Relational Database（PostgreSQL）
  - Graph Database（Neo4j）
  - Vector Database（pgvector）

### 資料處理

- 消化 JSON 資料並寫入資料庫
- JSON 的 `id` 不見得直接作為資料庫 PK，需**另外設計**
- 可以調整資料庫設計；Key 需要特別設計
- 易於維護：真實世界的 ID 不見得適合作為 FK/PK

### 向量資料庫

- 語言模型以向量表示文字
- 用於**向量搜尋（Vector Search）**
- 課程文件與期末專案摘要放入向量資料庫
- 接受「一個文件一個向量」的做法（即使真實世界通常切更細）

### 圖資料庫

- 設計**圖形資料庫**（Neo4j）
- 範例：學生 → 課程 → 主題（節點與關係）
- 可用於查詢「某課程涵蓋哪些 topic」

### 評分標準

- **重點在 Debug Panel**：讓語言模型能正確撈取資料庫資料
- **資料庫有正確抓取即算分**
- 語言模型本身的回應品質**不列入評分**

### 視覺化

- 需好好視覺化資料庫的**ER 圖 / 圖形結構**
